# 🤝 协作指南 — 文件即记忆（File as Memory）

> 本指南面向所有团队成员（人类 + AI Agent）。核心理念：**所有知识沉淀在 Markdown 文件中，Git 历史即团队记忆。**

---

## 核心理念

我们不依赖 Notion、Jira 或飞书做项目管理。一切信息保存在 Git 版本控制的 Markdown 文件中：

- **Git 历史** = 决策演变轨迹（为什么从 A 改到 B）
- **Markdown 文件** = 当前状态快照（团队现在在哪里）
- **GitHub PR** = 变更的上下文和讨论（为什么做这个变更）

好处：零工具成本、AI 可直接读写、离线可用、历史可回溯。

## 分支策略

```
main
 ├── feature/demo-page      ← 功能分支：一个具体功能
 ├── feature/ppt-outline
 ├── fix/data-pipeline
 └── temp/yang-draft        ← 临时草稿（temp_ 前缀）
```

### 规则

| 场景 | 分支名 | 说明 |
|------|--------|------|
| 主线 | `main` | 始终可展示的状态 |
| 新功能 | `feature/简述` | 从 main 切出，PR 合并 |
| 修复 | `fix/简述` | 从 main 切出，PR 合并 |
| 临时草稿 | `temp/姓名-内容` | 随意，不强制 PR |

- **禁止直接 push 到 main**（小修小补除外）
- 功能完成后通过 PR 合并，至少 1 人 review
- PR 描述写清楚"为什么做这个变更"

## Commit 规范

```
<type>(<scope>): <subject>

[可选：详细说明]
```

| Type | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 |
| `docs` | 文档更新 |
| `refactor` | 重构（无功能变化）|
| `chore` | 杂项（配置、依赖等）|

示例：
```
feat(demo): 添加候选人匹配界面

- 实现简历上传和解析
- 添加匹配度评分展示
```

## PR / Review 流程

1. **创建 PR**：从 feature 分支 → main
2. **PR 描述模板**：
   - 变更内容（What）
   - 原因（Why）
   - 如何验证（How to test）
3. **Review 要求**：
   - 至少 1 个 approve 才能合并
   - Review 意见直接写在代码行上
   - 有争议时线下对齐，线上只记结论
4. **合并后**：删除 feature 分支

## Issue 使用场景

| 场景 | 标签 | 说明 |
|------|------|------|
| 任务追踪 | `task` | 具体待办，可分配给某人 |
| 技术方案讨论 | `design` | 架构/方案选型 |
| Bug | `bug` | 问题记录 + 复现步骤 |
| 阻塞项 | `blocked` | 等外部输入才能推进 |

**原则**：能进 Markdown 文件的尽量不进 Issue。Issue 用于跨文件引用、需要多人异步讨论的问题。

## AI Agent 参与方式

### 新 AI 会话启动流程

每次开启新的 AI 会话时，告诉 AI：

> "读取 `AGENTS.md` 并执行唤醒协议"

AI 会自动按顺序读取：
1. `团队/成员与分工.md` — 了解团队成员
2. `团队/协作约定.md` — 了解沟通规则
3. `上下文/AI画像.md` — 了解团队风格
4. `上下文/会话日志.md` — 了解当前进度
5. `上下文/待办.md` — 了解当前任务
6. `上下文/决策.md` — 了解已做决定
7. `项目/2026-05-黑客松项目/README.md` — 了解项目约束

读取后，AI 会用 1-2 句话总结当前状态。

### 会话结束检查清单

会话结束前，提醒 AI：

> "请更新记忆文件：会话日志、待办、AI画像"

## 人类与 AI 的协作边界

| 工作 | 谁主导 | 谁协助 | 输出位置 |
|------|--------|--------|----------|
| 写代码 | AI | 人类 review | 代码文件 + PR |
| 写文档 | AI | 人类纠正 | Markdown 文件 |
| 做决策 | **人类** | AI 提供选项 | `上下文/决策.md` |
| 更新进度 | AI | 人类纠正 | `上下文/会话日志.md` |
| 分配任务 | **人类** | AI 建议优先级 | `上下文/待办.md` |
| 学习团队风格 | AI | 人类纠正 | `上下文/AI画像.md` |

**关键原则**：
- AI 不能替人类做决策（决策需要人类所有权）
- 人类通过编辑 Markdown 文件"告诉 AI 下一步做什么"
- 口头指令是临时的，文件指令是持久的

## 快速开始（新成员 / 新 AI）

### 人类新成员

1. 克隆仓库
2. 读 `AGENTS.md` 了解项目
3. 读 `团队/协作约定.md` 了解规则
4. 在 `团队/成员与分工.md` 中补充自己的信息
5. 看 `上下文/待办.md` 了解当前任务

### AI Agent

1. 读取 `AGENTS.md`
2. 执行唤醒协议（按顺序读取 7 个文件）
3. 总结当前状态
4. 等待人类在 `待办.md` 中分配任务，或响应人类口头指令

## 常见错误

| 错误 | 后果 | 修复 |
|------|------|------|
| 口头告诉 AI 任务，不写 `待办.md` | AI 忘记，下次重复问 | 重要任务写进 `待办.md` |
| 跳过 AGENTS.md 唤醒协议 | AI 不了解上下文，建议偏离 | 每次新会话先执行协议 |
| 决策后不更新 `决策.md` | 下次重复讨论同一个问题 | 决策后立即记录 |
| 把 AI 当搜索引擎 | 浪费上下文，得不到深度帮助 | 把 AI 当团队成员，提供完整上下文 |
